# digitakt-llm v1.0.0 — Design Spec

**Date:** 2026-04-12
**Status:** Approved

---

## Vision

v1.0.0 is a creative milestone: a tool you can play a 45-minute hybrid techno set with. Pre-composed patterns form the foundation; live generation handles variations and fills on top. The terminal feels like an instrument, not a developer tool.

**Definition of done:** you can play a 45-minute hybrid set — pre-composed library as foundation, live Claude generation for variations and fills — without the app feeling like a developer tool.

---

## Scope

Three milestones, each shippable independently:

| Milestone | Name | Delivers |
|---|---|---|
| v0.2 | Chain Engine | Pattern chaining, setlists, bar-boundary transitions |
| v0.3 | Instrument Panel | UI overhaul, persistent status bar, observability |
| v0.4 | Feedback Loop | Claude sees what's playing, section-aware generation |
| **v1.0** | **Integration** | All three combined, bug pass, README updated |

**Explicitly out of scope:** provider layer (Ollama/OpenAI), SysEx upload, error recovery overhaul, onboarding refactor.

## Milestone 1: Chain Engine (v0.2)

### Problem

`/load` queues one pattern at a time. A live set needs to think in sequences — `intro → build → drop → break → drop2` — and step through them at bar boundaries without interrupting flow.

### New State (`core/state.py`)

```python
chain: list[str] = field(default_factory=list)   # ordered pattern names
chain_index: int = -1                              # -1 = not in chain mode
chain_auto: bool = False                           # auto-advance vs. manual
```

### New Commands

| Command | Behavior |
|---|---|
| `/chain <p1> <p2> ...` | Define a setlist; names must exist in pattern library |
| `/chain --auto` | Auto-advance at bar boundary; loops at end |
| `/chain-status` | Show current position in chain |
| `/chain-next` | Queue the next pattern in chain at bar boundary |
| `/chain-clear` | Exit chain mode, clear chain state |

### New Backend Endpoints

| Endpoint | Behavior |
|---|---|
| `POST /chain` | Set chain (`names: list[str]`, `auto: bool`) |
| `POST /chain/next` | Advance to next in chain |
| `GET /chain` | Return current chain state |

### Implementation Notes

- `/chain-next` reuses the existing bar-boundary queued-swap logic from `/load` — `FillFSM` and `PatternMutator` are unchanged
- `/chain --auto` hooks into the bar-boundary event: when a pattern swap completes, if `chain_auto` is true, automatically calls `chain/next`
- Validation: all names in the chain must resolve to saved patterns at `/chain` time; unknown names return a 422 with the first unknown name identified
- WebSocket events: `chain_updated` (on `/chain`), `chain_advanced` (on each step through)

## Milestone 2: Instrument Panel (v0.3)

### Problem

The terminal UI is functional but reads like a developer tool. Mid-set, you can't tell at a glance what's queued, what Claude last did, or whether MIDI is healthy.

### Aesthetic Direction: Elektron-noir

The Digitakt has a dark, precise Swedish-industrial aesthetic. The TUI matches it: high information density, amber/orange status indicators, per-track color identity, no decorative chrome.

### Color Palette

| Role | Color | Usage |
|---|---|---|
| Active / hot | `#FF6B00` amber-orange | playhead, active step, current pattern in chain |
| Beat elements | `#E8E8E8` near-white | active step fill characters |
| Inactive | `#444` dark grey | empty steps, dim labels |
| MIDI ok | `#00FF88` green | connection dot |
| MIDI error | `#FF3366` red | connection dot |
| Claude idle | `#888` mid-grey | status dot |
| Claude busy | `#FFD700` gold | animated spinner dot |

### Track Color Map

```
BD  → #FF6B00  orange
SD  → #FF3366  rose
LT  → #FF9F1C  amber
MT  → #7B61FF  violet
HT  → #00D4FF  cyan
CL  → #00FF88  green
CH  → #FFD700  gold
OH  → #FF6BFF  magenta
```

### Step Grid Characters

| State | Char | Meaning |
|---|---|---|
| Active, full vel | `■` | solid block |
| Active, low vel | `▪` | small block |
| Active, conditional | `⊕` | circled plus |
| Active, low prob | `░` | light shade |
| Empty | `·` | mid-dot |
| Playhead on active | `▶` | amber, driven by MIDI clock |
| Playhead on empty | `▷` | dim, driven by MIDI clock |

### Layout

```
╔═ DIGITAKT-LLM ════════════════════════════════════════════════════════════╗
║  ▶ 140 BPM  SW:20  ●MIDI  ○Claude  ═══════════  [DROP1]  16 steps  bar:3 ║
╠═ CHAIN ════════════════════════════════════════════════════════════════════╣
║  intro ──→ build1 ──→ [DROP1] ──→ break ──→ drop2                        ║
╠═ PATTERN ══════════════════════════════════════════════════════════════════╣
║  BD  ▶■ · · · ■ · · · ■ · · · ■ · · ·                                   ║
║  SD  · · · · ■ · · · · · · · ■ · · ·                                     ║
║  HH  ■ · ■ · ■ · ■ · ■ · ■ · ■ · ■ ·                                   ║
║  ...                                                                       ║
╠═ LAST GEN ═════════════════════════════════════════════════════════════════╣
║  ↳ "darker, sparser kick"  →  kick×4  sd×2  hh×8  perc×1  ·  32ms       ║
╠═ LOG ══════════════════════════════════════════════════════════════════════╣
║  [12:04:33] pattern loaded: drop1                                          ║
║  [12:04:41] generating...                                                  ║
╚════════════════════════════════════════════════════════════════════════════╝
  BEAT > _
```

### Animation

- **Playhead** (`▶`/`▷`): moves across step grid in sync with MIDI clock via WebSocket `tick` events
- **Claude generating**: status dot cycles through `○◌◎●` at ~200ms interval
- **Queued chain advance**: upcoming pattern in chain pulses (alternates dim↔bright) when `/chain-next` is staged

### New Ink Components

| Component | File | Responsibility |
|---|---|---|
| `StatusBar` | `components/StatusBar.tsx` | BPM, swing, MIDI dot, Claude dot, pattern name, bar count |
| `ChainPanel` | `components/ChainPanel.tsx` | Setlist with position indicator; hidden when no chain defined |
| `StepGrid` | `components/StepGrid.tsx` | Replaces `PatternGrid`; per-track colors, playhead, character mapping |
| `GenerationSummary` | `components/GenerationSummary.tsx` | Last prompt + result stats in one compact line |
| `useClock` | `hooks/useClock.ts` | Drives playhead position from WebSocket tick events |

### WebSocket Events Required

- `tick` — current step index (drives playhead); backend must emit this at each MIDI clock step
- `chain_updated` — full chain state for `ChainPanel`
- `chain_advanced` — new index when chain steps forward
- `generation_summary` — `{ prompt, step_count, track_summary, latency_ms }` for `GenerationSummary`

## Milestone 3: Feedback Loop (v0.4)

### Problem

Claude is a one-shot generator with no awareness of what's playing. Mid-set, you have to re-describe the musical context from scratch on every prompt.

### Context Injection

The `/next`, `/vary`, and `/read` endpoints inject current state into the system prompt. Bare `/generate` (beat mode) is unchanged — no context injection, full user control.

```
[current context]
Pattern: drop1 (16 steps)
Active tracks: BD×4, SD×2, HH×8, OH×2
BPM: 140, swing: 20
Chain position: 3/5 (post-drop, heading toward break)
```

This is implemented in a new `build_context_prompt()` helper in `generator.py` that serializes the relevant fields from `AppState`.

### New Commands

| Command | Behavior |
|---|---|
| `/next [section]` | Claude generates the next pattern informed by current state + optional section hint (`drop`, `break`, `build`, `outro`) |
| `/vary [light\|medium\|heavy]` | Variation of current pattern at specified intensity — same energy envelope, mutated details |
| `/read` | Claude describes what's currently playing in musical terms |

### Distinction: `/next` vs. bare beat-mode prompt

| Bare beat prompt | `/next [section]` |
|---|---|
| Claude only knows your text | Claude knows current pattern + BPM + swing + chain position |
| No musical continuity | Generates with awareness of what came before |
| You describe what you want | You declare where you're going; Claude handles transition logic |

### New Backend Endpoints

| Endpoint | Behavior |
|---|---|
| `POST /next` | Generate next pattern with full context injection + optional `section: str` |
| `POST /vary` | Generate variation; accepts `intensity: "light" \| "medium" \| "heavy"` |
| `POST /read` | Return Claude's prose description of current pattern |

### Implementation Notes

- `build_context_prompt()` in `generator.py`: extracts `AppState` fields → formatted string → injected as system context prefix
- `/vary` intensity maps to a prompt instruction: `light` = "change at most 2 steps per track", `heavy` = "significantly restructure while keeping the same BPM feel"
- `/read` returns a `ChatResponse`-style payload, not a pattern — displayed in the activity log
- Bare beat-mode generation is unchanged; context injection is opt-in per endpoint

## What This Is Not

- Provider layer (Ollama/OpenAI) — deferred; Anthropic API is fine for live use
- SysEx upload — deferred per `DIRECTION.md`; USB MIDI is not a bottleneck
- Error recovery overhaul — deferred; not a live performance risk today
- Web or GUI frontend — never; the terminal is the instrument
